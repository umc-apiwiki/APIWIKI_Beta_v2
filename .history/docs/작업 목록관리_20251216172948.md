# 작업 목록 관리 가이드라인

PRD 완성 진행 상황을 추적하기 위한 마크다운 파일 작업 목록 관리 지침입니다.

---

## 작업 구현 규칙

### 작업 진행 원칙
- **한 번에 하나의 상위 작업만 수행**
  - 사용자에게 허가를 요청하고 yes 또는 y 응답을 받을 때까지 다음 상위 작업을 시작하지 않습니다.
  - 하위 작업의 경우 허락없이 계속 진행합니다.

### 완료 프로토콜
1. **하위 작업 완료 시**
   - 즉시 `[ ]`를 `[x]`로 변경하여 완료 표시.
   
2. **상위 작업 완료 시**
   - 모든 하위 작업이 `[x]`로 표시되면 상위 작업도 `[x]`로 표시.

3. **작업 후 절차**
   - 각 하위 작업 완료 후 중지하고 사용자 승인 대기.
   - 승인 후 Git 커밋 수행.

---

## Git 커밋 규칙

### 커밋 수행 시점
1. **하위 작업 완료 시**: 즉시 커밋 (필수).
2. **상위 작업 완료 시**: Git push 수행.

### 커밋 메시지 구조
```
type(task_num): Subject
body
footer
```

### Type 종류
- `feat`: 새로운 기능 추가.
- `fix`: 버그 수정.
- `docs`: 문서 수정.
- `style`: 코드 포맷팅, 세미콜론 누락 등 (코드 변경 없음).
- `refactor`: 코드 리팩토링.
- `test`: 테스트 코드 추가/수정.
- `chore`: 빌드 업무, 패키지 매니저 수정

### Subject 작성 규칙
- 최대 50글자.
- 마침표 사용 안 함.
- 영문 작성 시: 동사 원형으로 시작, 첫 글자 대문자.
- 가능한 한국어로 작성할 것

### Body 작성 규칙
- 긴 설명이 필요한 경우 작성.
- 어떻게가 아닌 무엇을, 왜 했는지 작성.
- 최대 75자 이내.
- 영어로 작성할 것

### Footer 작성 규칙
- Issue tracker ID 또는 Task 정보 명시.
- 형식: `유형: #이슈번호`.
- 여러 이슈는 쉼표(,)로 구분.

**Footer 유형:**
- `Fixes`: 이슈 수정 중 (미해결).
- `Resolves`: 이슈 해결 완료.
- `Ref`: 참고 이슈.[8]
- `Related to`: 관련 이슈 (미해결).

**예시:**
```
fix(task 2.1.2): Implement login functionality

Add JWT-based authentication flow

Resolves: #42
```

## 작업 목록 유지관리

### 작업 목록 업데이트
- 프로토콜에 따라 작업 완료 표시 (`[x]`).
- 새로운 작업 발견 시 즉시 추가.

### 관련 파일 섹션 관리
- 생성/수정된 모든 파일 나열.
- 각 파일에 목적 한 줄 설명 작성.

### 작업 목록 계층 구조
```markdown
- [ ] 1.0 상위 작업
    - [ ] 1.1 하위 작업
        - [ ] 1.1.1 테스트 코드 작성
        - [ ] 1.1.2 테스트 실행 및 검증
        - [ ] 1.1.3 오류 수정 (필요 시)
    - [ ] 1.2 하위 작업
        - [ ] 1.2.1 테스트 코드 작성
        - [ ] 1.2.2 테스트 실행 및 검증
        - [ ] 1.2.3 오류 수정 (필요 시)
```

## AI 실행 지침

작업 목록 관리 시 AI는 다음을 반드시 수행합니다.

### 즉시 체크
- 완료한 작업은 즉시 체크 표시.[6]

### 정기 업데이트
- 중요 작업 완료 후 작업 목록 파일 업데이트.

### 완료 프로토콜 준수
- 완료된 하위 작업을 `[x]`로 표시.[6]
- 모든 하위 작업 완료 시 상위 작업도 `[x]`로 표시.

### 작업 목록 동적 관리
- 새로 발견된 작업 추가.[2][5]
- 필요 시 작업 목록 생성/수정.[1]
- **모든 하위 작업마다 그 하위에 테스트 관련 작업 목록 생성:**
  - `[번호].1` 테스트 코드 작성
  - `[번호].2` 테스트 실행 및 검증
  - `[번호].3` 오류 수정 (필요 시)

### 테스트 오류 처리
- 테스트 진행 중 오류 발생 시.[3]
- 해당 하위 작업의 하위 레벨에 오류 수정 작업 추가.
- 예: `1.1.3.1`, `1.1.3.2` 형식으로 세부 오류 수정 작업 생성.

### 관련 파일 최신 상태 유지
- 정확하고 최신의 파일 목록 관리.[10]

### 작업 시작 전 확인
- 다음 하위 작업이 무엇인지 확인.[13]

### 작업 완료 후 절차
- 파일 업데이트.[5]
- 사용자 승인 대기.[13]

        - [ ] 2.1.1 테스트 코드 작성

## 작업 흐름 요약

```
작업 시작 → 하위 작업 확인 → 작업 수행 → 테스트 코드 작성 → 
테스트 실행 → (오류 시) 오류 수정 → 작업 목록 체크 → 
Git 커밋 → 사용자 승인 대기 → (승인 시) 다음 작업 진행
```

**상위 작업 완료 시**: Git Push 수행.
